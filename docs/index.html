Glance Dashboard Documentation

Your complete guide to setting up, configuring, and using the Glance real-time telemetry visualization platform. Built by Team Ignition.

Overview

Glance is a professional-grade, real-time telemetry visualization platform designed for rockets, drones, embedded systems, and advanced data-acquisition applications. Built with PySide6 (Qt6), it offers a modular, widget-based dashboard that adapts to diverse telemetry needs. With multi-source connectivity, powerful data logging, and an intuitive UI/UX, Glance empowers engineers, researchers, and operators to gain actionable insights from complex data streams—all at a single glance.

Primary Goal: To provide a powerful, flexible, and user-friendly tool for monitoring live data, enabling rapid diagnostics, performance analysis, and decision-making during critical operations.

Installation Guide

1. Clone the Repository

First, clone the project from GitHub to your local machine.

bash 
git clone https://github.com/teamignitionvitc/Glance.git
cd Glance

2. Set Up a Virtual Environment (Recommended)

Using a virtual environment prevents conflicts with other Python projects.

bash 
python -m venv venv
# On Windows: venv\Scripts\activate
# On macOS/Linux: source venv/bin/activate

3. Install Dependencies

Install all required libraries using the requirements.txt file.

bash 
pip install -r requirements.txt

For the Map Widget and PDF Reporting, optional dependencies are required. Install them with: pip install PySide6-WebEngine reportlab

Coming Soon: We are actively working on one-click executables for Windows (.exe) and Linux (.AppImage) to make installation even easier. These will be available in upcoming releases!

4. Launch the Application

Once dependencies are installed, you can start Glance.

bash 
python main.py
For more details, see the Installation section in the README.

The Glance Workflow

Glance guides you from a blank slate to a fully functional dashboard in four clear phases.

1

Phase 1: Welcome

The initial screen provides two main paths: creating a new dashboard from scratch or loading a previously saved project configuration file (.json). This is your starting point for any session.

2

Phase 2: Connection & Data Setup

Here, you tell Glance where your data is coming from and how it's formatted. You'll select a Data Source (like Serial or TCP) and specify the structure of the incoming data packets. This is crucial for a successful connection.

3

Phase 3: Initial Widget Layout

After setting up the data source, you can pre-configure the widgets you want to see. Select a widget type, link it to one or more defined Parameters, and add it to the initial layout. You can also skip this and add widgets manually later.

4

Phase 4: The Live Dashboard

This is the main interface where you monitor your data in real-time. It's a fully interactive environment where you can drag, drop, resize, and float widgets to create a layout that suits your needs.

Connecting to Data Sources

Glance can connect to a variety of hardware and software data sources. This is configured in the setup wizard or via File → Connection Settings.... If you encounter issues, see the Troubleshooting section.

Serial

Used for direct hardware connections via USB-to-Serial adapters, RS232, or on-board USB ports (like an Arduino or ESP32). You must specify the correct port name and baud rate.

  • Finding the Port: The port name depends on your operating system.
    • Windows: The port is named COM followed by a number (e.g., COM3, COM11). You can find it in the "Ports (COM & LPT)" section of the Device Manager.
    • Linux: The port is typically named /dev/ttyUSB0 for USB-to-Serial adapters or /dev/ttyACM0 for devices like Arduino. You can list available ports by running ls /dev/tty* in a terminal.
    • macOS: The port name usually starts with /dev/cu. (e.g., /dev/cu.usbserial-A50285BI or /dev/cu.usbmodem14101).
  • Baud Rate: This must exactly match the baud rate configured on your device (e.g., Serial.begin(115200) in Arduino code). Common values are 9600, 57600, and 115200.

TCP

Used for receiving data over a network from a device or application acting as a server. Glance acts as the client and connects to the server's IP address and port.

  • Host: The IP address of the device sending the data (e.g., 192.168.1.101).
  • Port: The port number the server application is listening on (e.g., 9000).
  • Common Pitfall: Firewalls on either the sending or receiving machine can block TCP connections. Ensure the port is open.

UDP

Used for high-frequency data streaming where the occasional loss of a packet is acceptable. Glance acts as a listener on a specific port.

  • Host: Typically set to 0.0.0.0 to listen for packets from any device on the network.
  • Port: The port number Glance will listen on for incoming data. The sending device must be configured to send UDP packets to the IP address of the computer running Glance, on this specific port.

Dummy

A built-in data generator for testing and development. It produces a configurable number of channels with sine wave data and random noise, requiring no external hardware.

For more details, see the Data Sources section in the README.

Configuring Data Formats

Telling Glance how to interpret the incoming stream of bytes is one of the most critical configuration steps. An incorrect data format setting will result in garbled or no data appearing on your widgets.

JSON Array (json_array)

This is the simplest and most human-readable format. Glance expects a standard JSON array of numbers for each packet.

  • Structure: A text string starting with [ and ending with ], with numbers separated by commas.
  • Example Packet: [25.5, 55.2, 1013.25, 9.81, -0.5, 0.2]
  • Required Settings:
    • Channel Count: Must match the number of elements in your array.

CSV (csv)

This format uses a delimiter (like a comma) to separate values. It's less overhead than JSON but still text-based.

  • Structure: A text string of numbers separated by a specific character.
  • Example Packet: 25.5,55.2,1013.25,9.81,-0.5,0.2
  • Required Settings:
    • Channel Count: Must match the number of values.
    • CSV Separator: The character used to separate values (e.g., ,, ;, |).

Raw Bytes (raw_bytes)

This is the most efficient format, sending raw binary data without any text formatting. It requires precise configuration to be read correctly.

Warning: For raw_bytes, all settings—Channel Count, Sample Width, and Endianness—must exactly match the configuration of the device sending the data.

  • Channel Count: The total number of distinct values in one complete data packet.
  • Sample Width (Bytes): The number of bytes used for each single value. For example, a 16-bit integer uses 2 bytes; a 32-bit float uses 4 bytes.
  • Endianness: This defines the order of bytes for numbers larger than one byte.
    • Little-Endian: The least significant byte is sent first. This is common for Intel/AMD processors and many microcontrollers (e.g., Arduino, ESP32).
    • Big-Endian: The most significant byte is sent first. This is common in networking protocols and older architectures.

Example: Your device sends 3 channels of data as 16-bit (2-byte) little-endian integers. Your total packet size is 3 channels * 2 bytes/channel = 6 bytes.

If the values are `[258, 515, 772]`, their hexadecimal representations are `[0x0102, 0x0203, 0x0304]`. The byte stream Glance expects to receive would be:

hex
02 01 03 02 04 03
For more details, see the Data Formats section in the README.

Managing Parameters

Parameters are the heart of Glance. They act as translators, mapping raw incoming data (like a value at a specific position in an array) to a meaningful, human-readable channel that widgets can display. You can configure them via Edit → Manage Parameters.

The Array Index is Critical: This zero-based number tells Glance where to find the data for this parameter in the incoming data packet. An incorrect index is the most common reason for data not appearing on a widget.

Key Parameter Properties:

PropertyDescription
ID & NameA unique programmatic ID (no spaces) and a human-readable display name.
Array IndexThe position (0, 1, 2, etc.) of this parameter's value within the incoming data packet.
UnitThe measurement unit to display (e.g., °C, m/s², V).
ThresholdsFour values (Low Critical, Low Warning, High Warning, High Critical) that define alarm states and color-coding in widgets. Must be in ascending order.
Sensor GroupAn optional text field for organizing parameters (e.g., "IMU", "Power System").
DescriptionAn optional field for more details about the parameter.
For more details, see the Parameters section in the README.

The Widget Library & Customization

Widgets are the visual components of your dashboard. Add them by clicking "Add Widget..." and linking them to one or more configured parameters. Each widget has unique behaviors and customization options derived from the parameter settings.

Value Card

Displays a single parameter's value in a large, easy-to-read format. Its appearance is highly dynamic based on parameter settings.

  • Alarm State: The color of the value text changes based on where the current value falls within the defined thresholds:
    • Nominal: White text.
    • Warning: Yellow text.
    • Critical: Red text.

Time Graph

A powerful multi-parameter line chart for trend analysis. It is highly interactive.

  • Interactivity: Features interactive zoom (mouse wheel), pan (click-and-drag), and a crosshair for inspecting values on hover.
  • Data Point Inspection: Clicking on the graph near a data point will show a dialog with the precise value and timestamp for that point.
  • Colors: Line colors are automatically assigned from a predefined palette to ensure contrast.

Gauge

A classic circular gauge that shows a value within a defined range. The parameter's thresholds are used to draw colored arcs on the gauge, providing an immediate visual sense of the value's state relative to its operational limits.

LED Indicator

A simple status light that provides a quick, at-a-glance indication of a parameter's state based on its thresholds.

  • Blue: Value is below the 'Low Warning' threshold.
  • Green: Value is in the normal range (between 'Low Warning' and 'High Warning').
  • Yellow: Value is in the high warning range (between 'High Warning' and 'High Critical').
  • Red: Value is in the critical range (below 'Low Critical' or above 'High Critical').

Log Table

Displays a timestamped, scrolling table of raw data values for multiple parameters. Includes a powerful search function to find the last occurrence of a value that matches a condition (e.g., `> 100`) and highlights the corresponding row.

Histogram

Visualizes the statistical distribution of a single parameter over time, showing how frequently different value ranges occur. Useful for analyzing sensor noise and data patterns.

Map (GPS)

Plots latitude and longitude coordinates on a satellite map. It requires exactly two parameters (latitude and longitude) to function. It shows the current position with a marker, draws a path of previous locations, and includes a button to zoom directly to the latest point.

Note: The Map widget requires the PySide6-WebEngine optional dependency to be installed.

For more details, see the Widget Types section in the README.

Data Logging & Summary Reports

Glance provides powerful tools for recording and analyzing your telemetry sessions. You can access all logging features from the Data Logging menu.

Configuring and Recording a Session

To record a session, first go to Data Logging → Configure Logging.... Here you can select CSV or JSON as the output format, choose which specific parameters to log, and set a buffer size.

Understanding the Buffer

Data is temporarily stored in memory (the buffer) before being written to the disk. This improves performance by reducing constant disk access.

  • A larger buffer means fewer disk writes, which is better for performance with high-frequency data, but risks more data loss if the application crashes.
  • A smaller buffer writes to the disk more often, which is safer but can impact performance at very high data rates.

Once configured, you can start and stop logging at any time using the menu actions or the shortcuts Ctrl+Shift+L and Ctrl+Alt+L.

Generating a Summary Report

After you have stopped a logging session, you can instantly generate a professional PDF summary of the collected data. This is an excellent tool for post-mission analysis, sharing results, and archiving session statistics.

  • How to Access: Navigate to Data Logging → Generate Summary Report... or use the shortcut Ctrl+Shift+R.
  • Prerequisites: You must have a completed log file from the current session, and the reportlab Python library must be installed.

Installation: If you don't have it, install the reporting library with: pip install reportlab

Report Contents

The generated PDF is a multi-page document that includes:

  1. Report Information: Metadata about the session, including the source file, duration, total samples, and start/end times.
  2. Parameter Statistics: A summary table showing the min, max, mean, and sample count for every parameter that was logged.
  3. Detailed Parameter Analysis: A breakdown for each individual parameter with more detailed statistics like range and standard deviation.
For more details, see the Data Logging section in the README.

Advanced Tools

Raw Telemetry Monitor

Accessible from the View menu or the status bar, the Raw Telemetry Monitor is a powerful debugging tool. It displays the raw, unparsed data packets exactly as they are received by the application, along with timestamps and statistics. Use this monitor to:

  • Verify that data is being received from your source.
  • Inspect the structure of your data packets to ensure it matches your configured format.
  • Switch between Decimal, Hex, ASCII, and Binary views to debug data.
  • Pause, clear, and save the raw log for later analysis.
For more details, see the Monitoring & Debugging section in the README.

Signal Filtering & Processing

Glance includes a powerful real-time signal filtering system to help you reduce noise, smooth out sensor readings, and remove outliers directly within the application. Filters are applied non-destructively, meaning the raw data is always preserved for logging, while widgets display the cleaned-up, filtered values for clearer visualization.

You can manage filters from the main menu under Filters. You can add a new filter to any defined parameter or open the Manage Filters dialog to view, enable/disable, or remove all active filters for all parameters.

Available Filter Types

Each filter is designed for a specific purpose. You can even chain multiple filters on a single parameter for advanced processing.

Filter Type Best For Description & Key Parameter(s)
Moving Average General purpose smoothing A simple filter that smooths data by averaging the last 'N' data points. It's effective for reducing random noise but can introduce a slight lag.
  • Window Size: The number of recent data points (N) to include in the average. A larger window provides more smoothing.
Low Pass Filter Continuous smoothing An efficient exponential moving average (EMA) that gives more weight to recent data points. It provides very smooth output and is great for signals that don't change abruptly.
  • Alpha (α): The smoothing factor from 0.01 to 1.0. A lower value results in more smoothing but a slower response to real changes in the signal.
Median Filter Removing outliers & spikes This filter is excellent for removing sharp, short-duration spikes (outliers) from a signal. It works by replacing each data point with the median value of its neighbors, effectively ignoring extreme, isolated values.
  • Window Size: The number of points to consider for the median calculation. An odd number (e.g., 3, 5, 7) is recommended.
Kalman Filter Noisy sensors An advanced, adaptive filter ideal for noisy sensors. It works by predicting the next state of the system and then correcting its prediction based on the actual measurement. It's highly effective at producing a smooth, accurate estimate from an erratic signal.
  • Process Variance (Q): How much you expect the signal to change between readings. A higher value means the filter adapts more quickly.
  • Measurement Variance (R): How much noise you expect in your sensor readings. A higher value means you trust the measurements less.
For more details, see the Signal Filtering section in the README.

Development Guide

Glance is an open-source project driven by community collaboration. This guide provides the technical details needed to understand the codebase, add new features, and contribute to the project.

Architectural Overview

The application is built on the Qt framework via PySide6 and leverages its powerful signals and slots mechanism for communication between components. The architecture follows a Model-View-Controller-like pattern:

  • Model: The data acquisition layer (app/backend.py) and the application's state (parameters, widget configs, data history) managed by the `MainWindow`.
  • View: The various widget classes (app/widgets.py) and dialogs (app/dialogs.py) that present data and options to the user.
  • Controller: The `main.py` file containing the `MainWindow` class, which connects the data backend to the UI, handles user input, and orchestrates the overall application logic.

Project Code Structure

The project is organized into modules with specific responsibilities.

bash 
.
├── main.py              # Application entry point, MainWindow class, all dialogs and widgets.
├── backend.py           # DataReader class for Serial, TCP, UDP sources.
├── docs/                # Documentation files and images for the web guide.
├── logs/                # Default directory for saved data logs.
└── requirements.txt     # Python dependencies.

Core Data Flow

Understanding the data flow is key to modifying or extending Glance.

  1. The DataSimulator (or `DataReader` via `backend.py`) runs in a separate QThread.
  2. Upon receiving/generating a complete data packet, it emits a newData signal with the data (as a list of values).
  3. The MainWindow has a slot, update_data(packet), connected to this signal.
  4. Inside update_data, the `MainWindow` iterates through its list of defined parameters. For each parameter, it uses the array_index to get the corresponding value from the packet.
  5. The value is passed through the FilterManager, which applies any configured filters (e.g., Low-pass, Kalman).
  6. The raw value and filtered value are appended to the global data_history dictionary.
  7. The `MainWindow` then identifies which active widgets are subscribed to the updated parameter and calls their respective update methods with the filtered value.

Adding a Custom Widget

Follow these steps to create and integrate a new widget type:

  1. Create the Widget Class in main.py:

    Your class must inherit from a Qt widget (like QWidget or QFrame) and have an update method.

  2. Register in AddWidgetDialog:

    In the `AddWidgetDialog` class, add the name of your new widget to the `display_types` list to make it selectable by the user.

  3. Add Handler in MainWindow.add_widget_to_dashboard:

    In this method, add a new `elif` block to instantiate your widget class when its display type is chosen.

Building an Executable

To create a standalone executable, you can use PyInstaller. This packages your application and its dependencies into a single file.

bash 
# Install PyInstaller
pip install pyinstaller

# Build the executable from the project root directory
pyinstaller --onefile --windowed --name "Glance" \
--icon="docs/public/Glance_nobg.ico" \
--add-data "docs/public:docs/public" \
main.py

Note: The --add-data flag is crucial for bundling assets like images. The syntax is "source:destination_in_bundle". You may need to adjust paths depending on your OS.

Testing Guide

Glance includes a comprehensive test suite to ensure code quality, reliability, and correctness. We use the pytest framework to write and run our unit, integration, and performance tests.

Test Structure

The tests are located in test.py and are organized into classes, with each class focusing on a specific component of the application. This includes dedicated tests for the data logger, signal filters, UI widgets, dialogs, and the main application logic.

To run the tests, you'll need pytest and pytest-cov. They are included in requirements.txt, but you can also install them manually: pip install pytest pytest-cov

Running the Tests

All test commands should be run from the root directory of the project.

Run All Tests

This command discovers and runs all tests in the test.py file.

bash 
pytest test.py -v

Run a Specific Test Class

To run all tests within a specific class (e.g., TestDataLogger), use the :: syntax.

bash 
pytest test.py::TestDataLogger -v

Run Tests Matching a Keyword

The -k flag allows you to run tests whose names contain a specific keyword (e.g., "filter").

bash 
pytest test.py -k "filter" -v

Generate a Coverage Report

To measure how much of the application code is covered by tests, use pytest-cov. This command runs tests and generates an HTML report in a new htmlcov/ directory.

bash 
pytest test.py --cov=main --cov-report=html

Troubleshooting FAQ

Here are solutions to some of the most common issues encountered during setup and operation.

My Serial Port is not in the list.

If the port for your connected device doesn't appear in the dropdown, try these steps:

  1. Refresh the List: Click the "Refresh" icon next to the dropdown menu to force Glance to scan for new ports.
  2. Check Drivers: Ensure you have the correct drivers installed for your USB-to-Serial adapter (e.g., FTDI, CH340).
  3. Verify Connection: Make sure the USB cable is fully plugged in and is a data cable, not just a charging cable.
  4. Check OS:
    • Windows: Open Device Manager and look under "Ports (COM & LPT)". If there's a yellow warning icon, the driver may be missing or incorrect.
    • Linux: You may need to add your user to the dialout group. Run sudo usermod -a -G dialout $USER, then log out and log back in.

I'm connected, but no data is appearing on my widgets.

This is a common issue with several potential causes. Follow this diagnostic checklist:

  1. Use the Raw Telemetry Monitor: Go to View → Raw Telemetry Monitor. This is the most important step. If you see data here, the problem is with your parsing configuration (step 2). If you don't see any data, the problem is with your physical connection or data source (step 3).
  2. Check Parsing and Parameter Settings:
    • Is the Data Format (JSON, CSV, Raw Bytes) set correctly to match what your device is sending?
    • Is the Channel Count correct?
    • Most importantly, is the Array Index for your parameter correct? Remember it's zero-based.
    • If using Raw Bytes, are the Sample Width and Endianness an exact match with your device?
  3. Check Connection Settings:
    • For Serial, is the Baud Rate identical to the one set on your hardware?
    • Are the TX and RX lines swapped? Try switching them.
    • For TCP/UDP, are the IP address and Port correct? Check for typos.
    • Is a firewall on either the sending or receiving computer blocking the connection?
  4. Check Stream Status: Ensure the stream is not paused. The status label in the top bar should say "STREAMING".

The data on my widgets is garbled or incorrect.

This almost always indicates a mismatch between the data being sent and how Glance is configured to parse it.

  • Baud Rate Mismatch: If using a serial connection, receiving at a different baud rate than the sender will result in meaningless characters. Verify the rates match.
  • Incorrect Data Format: If your device sends CSV but Glance expects JSON, it won't be able to parse it. Use the Raw Telemetry Monitor to see the exact format of the incoming text.
  • Raw Bytes Mismatch: If using `raw_bytes`, this is the most likely cause. A single incorrect setting (Sample Width, Channel Count, or Endianness) will corrupt the interpretation of all subsequent data. Double-check these against your device's firmware.

The application is running slow or lagging.

High data rates combined with many complex widgets can consume significant CPU resources. Try these optimizations:

  • Reduce Widget Complexity: Time Graphs are more resource-intensive than Value Cards or Gauges. Temporarily close complex widgets to see if performance improves.
  • Increase Logging Buffer: If you are logging data at a high frequency, go to Data Logging → Configure Logging... and increase the buffer size (e.g., to 500 or 1000). This reduces the frequency of disk writes.
  • Reduce Data Rate: If possible, lower the rate at which your hardware sends data. A rate of 10-50 Hz is often sufficient for visual monitoring.
